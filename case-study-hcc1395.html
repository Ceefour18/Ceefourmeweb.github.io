Flagship Case Study

HCC1395 Somatic WGS Pipeline

End-to-end tumour-normal whole genome sequencing pipeline for breast cancer, benchmarked against the SEQC2 gold-standard truth set — from raw FASTQ through Mutect2 calling, debugging, and ROC validation.

0.974 ROC AUC (SEQC2 validation)
~3× Faster via Colab migration
16 vCPU AWS pipeline
7 Automated stages

Problem

Clinical genomics teams need reproducible somatic variant calling pipelines that can be validated against community gold standards — not just "it runs." The SEQC2 consortium established HCC1395 (triple-negative breast cancer) and HCC1395BL (matched normal) as reference samples with high-confidence somatic SNV/INDEL truth sets across ~40,000 SNVs and complex genomic alterations.

The goal: build a Docker-based WGS pipeline targeting clinically relevant breast cancer genes, validate calls against the SEQC2 v1.2 truth set, and document the full debugging process — the kind of work expected in industry precision oncology platforms.

Approach

I implemented a modular bash-orchestrated pipeline deployed on AWS (r6i.4xlarge: 16 vCPU, 128 GB RAM, 500 GB gp3 storage), with each stage containerised for reproducibility:

  • Alignment — BWA-MEM on paired-end tumour/normal FASTQs with proper read groups
  • Pre-processing — GATK MarkDuplicates and BQSR against dbSNP138 + Mills indels
  • Calling — Mutect2 tumour-normal paired mode → FilterMutectCalls
  • Downstream — Structural variant calling and SnpEff functional annotation modules
  • Validation — ROC analysis against SEQC2 high-confidence regions; variant-level sensitivity/precision profiling

When local WSL2 compute became a bottleneck (~48 h end-to-end), I migrated the alignment and Mutect2 stages to Google Colab Pro+ with mounted Drive storage, cutting wall-clock runtime to ~14 h (~3× improvement) without changing the analytical logic.

Tools & Infrastructure

GATK 4.6 Mutect2 BWA-MEM SAMtools Docker AWS EC2 Google Colab SnpEff Python / R SEQC2 truth set GRCh38

Key Challenge: FASTQ Read-Name Mismatch

How I debugged a silent alignment failure

BWA-MEM alignment initially produced empty BAMs with no mapped reads. FastQC and head inspection of the FASTQ headers revealed the root cause: tumour R1 and R2 files used inconsistent read-name suffixes (e.g. /1 vs /2 mismatches, and truncated IDs between the SEQC2 SRA download and my local rename step). BWA requires strictly paired read names — a mismatch means zero proper pairs and a pipeline that "completes" but produces no callable variants.

Fix: I wrote a normalisation script to strip platform-specific suffixes and enforce consistent @READ_ID 1:N:0:... / @READ_ID 2:N:0:... naming across all four FASTQ files, then re-ran FastQC to confirm paired-end integrity before re-submitting alignment. This single fix unblocked the entire downstream Mutect2 chain — a concrete example of the read-level QC debugging that production pipelines depend on.

Results

  • ROC AUC = 0.974 on hold-out validation against the SEQC2 high-confidence somatic SNV truth set in HC regions
  • Pipeline validated with sensitivity/precision profiling against SEQC2 v1.2 PASS-labelled variants in benchmark regions
  • Full tumour-normal WGS workflow automated from FASTQ → filtered VCF with reproducible Docker mounts
  • Runtime reduced ~3× (48 h → ~14 h) by migrating compute-heavy stages to Colab Pro+
  • Complementary WES pipeline (HCC1143/chr17) independently recovered clonal TP53 p.Arg248Gln at 98.6% VAF with 1.11% cross-sample contamination
